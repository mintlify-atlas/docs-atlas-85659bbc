## Overview

Nodes are the building blocks of Flowbaker workflows. Each node represents a discrete operation, whether it's triggering on an event or performing an action.

## Node Types

Flowbaker has two fundamental node types:

```go
type NodeType string

const (
    NodeTypeTrigger NodeType = "trigger"
    NodeTypeAction  NodeType = "action"
)
```

Location: `pkg/domain/workflow.go:93`

### Trigger Nodes

Trigger nodes initiate workflow execution when specific events occur:

- **Webhook**: HTTP request received
- **Cron**: Scheduled time reached
- **Polling**: Data change detected
- **On Error**: Error occurred in another workflow
- **Chat**: User message received
- **Click**: Button clicked

<Tip>
Every workflow must have at least one trigger node to initiate execution.
</Tip>

### Action Nodes

Action nodes perform operations and transformations:

- **HTTP Requests**: Call external APIs
- **Database Operations**: Query or modify databases
- **Data Transformation**: Transform, filter, or aggregate data
- **AI Operations**: Call LLM models, use AI agents
- **Integrations**: Interact with third-party services
- **Logic**: Conditional branching, loops
- **Response**: Send response back to caller

## Node Inputs

Nodes receive data through inputs:

```go
type NodeInput struct {
    InputID          string
    SubscribedEvents []string
}
```

Each input:
- Has a unique `InputID` (e.g., `"input-node123-0"`)
- Subscribes to output events from upstream nodes
- Can receive data from multiple upstream nodes

Location: `pkg/domain/workflow.go:138`

### Multiple Inputs

Nodes can have multiple inputs:

```go
node := WorkflowNode{
    ID: "merge-node",
    Inputs: []NodeInput{
        {InputID: "input-merge-node-0", SubscribedEvents: []string{"output-node1-0"}},
        {InputID: "input-merge-node-1", SubscribedEvents: []string{"output-node2-0"}},
    },
}
```

<Note>
Nodes with multiple inputs wait for **all** inputs to receive data before executing (unless they're AI Agent nodes which handle inputs differently).
</Note>

Nodes produce outputs that flow to downstream nodes:

- **Output Format**: `output-{nodeID}-{index}`

## Node Settings

Each node has configurable settings:

```go
type NodeSettings struct {
    ReturnErrorAsItem       bool
    OverwriteExecutionLimit bool
    ExecutionLimit          int
}
```

### Setting Details

**ReturnErrorAsItem**
- When `true`: Errors are converted to items and passed downstream
- When `false`: Errors halt workflow execution
- Useful for error handling workflows

**OverwriteExecutionLimit**
- Override the workflow-level execution limit
- Allows per-node execution limits
- Prevents infinite loops in specific nodes

**ExecutionLimit**
- Maximum times this node can execute in one workflow run
- Only used if `OverwriteExecutionLimit` is `true`
- Default workflow limit is 1000

## Node Execution Flow

When a node executes:

### 1. Trigger Node Execution

```go
func (w *WorkflowExecutor) ExecuteTriggerNode(
    ctx context.Context, 
    node domain.WorkflowNode, 
    execution NodeExecutionTask,
) (NodeExecutionResult, error) {
    inputID := fmt.Sprintf(InputHandleFormat, node.ID, 0)
    inputPayload, exists := execution.PayloadByInputID[inputID]
    if !exists {
        return NodeExecutionResult{}, fmt.Errorf("trigger input payload not found")
    }
    
    return NodeExecutionResult{
        Output: domain.IntegrationOutput{
            ResultJSONByOutputID: []domain.Payload{inputPayload.Payload},
        },
        IntegrationType:       domain.IntegrationType(node.Type),
        IntegrationActionType: domain.IntegrationActionType(node.TriggerNodeOpts.EventType),
    }, nil
}
```

Trigger nodes simply pass through their input payload.

Location: `pkg/domain/executor/workflow_executor.go:444`

### 2. Action Node Execution

```go
func (w *WorkflowExecutor) ExecuteActionNode(
    ctx context.Context, 
    node domain.WorkflowNode, 
    execution NodeExecutionTask,
) (NodeExecutionResult, error) {
    // 1. Skip non-workflow nodes (e.g., agent tools)
    if node.UsageContext != "" && node.UsageContext != "workflow" {
        return NodeExecutionResult{/* empty output */}, nil
    }
    
    // 2. Select integration creator
    integrationCreator, err := w.integrationSelector.SelectCreator(ctx, 
        domain.SelectIntegrationParams{
            IntegrationType: node.IntegrationType,
        })
    
    // 3. Get credential ID from settings
    credentialID := node.IntegrationSettings["credential_id"]
    
    // 4. Create integration instance
    integrationExecutor, err := integrationCreator.CreateIntegration(ctx, 
        domain.CreateIntegrationParams{
            WorkspaceID:  w.workflow.WorkspaceID,
            CredentialID: credentialIDString,
        })
    
    // 5. Execute integration with inputs
    output, err := integrationExecutor.Execute(ctx, domain.IntegrationInput{
        NodeID:           node.ID,
        PayloadByInputID: payloadByInputID,
        Workflow:         &w.workflow,
        IntegrationParams: domain.IntegrationParams{
            Settings: node.IntegrationSettings,
        },
        ActionType: node.ActionNodeOpts.ActionType,
    })
    
    return NodeExecutionResult{
        Output:                output,
        IntegrationType:       node.IntegrationType,
        IntegrationActionType: node.ActionNodeOpts.ActionType,
    }, nil
}
```

## Expression Support

Node properties support expressions for dynamic values:

```go
type WorkflowNode struct {
    ExpressionSelectedProperties []string
    // ...
}
```

Expressions allow:
- Referencing upstream node outputs
- Dynamic value computation
- Conditional logic
- Data transformation

**Example Expression**:
```javascript
{{ $node.httpRequest.output.data.user.email }}
```

## Usage Contexts

Nodes can serve different purposes based on context:

```go
type ActionUsageContext string

const (
    UsageContextWorkflow       ActionUsageContext = "workflow"
    UsageContextTool           ActionUsageContext = "tool"
    UsageContextLLMProvider    ActionUsageContext = "llm_provider"
    UsageContextMemoryProvider ActionUsageContext = "memory_provider"
)
```

- **workflow**: Normal workflow node (executes)
- **tool**: AI agent tool (available to agent)
- **llm_provider**: LLM configuration (used by agent)
- **memory_provider**: Memory configuration (used by agent)

## Error Handling in Nodes

When a node encounters an error:

### Default Behavior

1. Error is logged
2. Workflow execution halts
3. Error event is published
4. Execution marked as failed

### With ReturnErrorAsItem

```go
type ErrorItem struct {
    ErrorMessage string `json:"error_message"`
}
```

1. Error is wrapped in an ErrorItem
2. Error item is passed to downstream nodes
3. Workflow continues execution
4. Downstream nodes can handle the error

## Best Practices

<Check>
**Do's**
- Use descriptive node names
- Set execution limits on loop nodes
- Enable ReturnErrorAsItem for error handling nodes
- Use expressions for dynamic configuration
- Configure proper credential IDs
</Check>

<Warning>
**Don'ts**
- Don't create circular dependencies without limits
- Don't hardcode sensitive data in settings
- Don't ignore node execution errors
- Don't connect incompatible node types
</Warning>
